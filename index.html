Travaux Pratiques

Cette section regroupe les TP réalisés en Web sémantique. L’objectif n’est pas seulement de déposer les comptes rendus PDF : je résume aussi les notions importantes, les fichiers produits et ce que chaque TP m’a apporté.

Fil conducteur

Le portfolio suit une progression : structurer une donnée avec XML, la valider avec DTD/XSD, la transformer avec XSLT, puis la relier avec RDF, SPARQL et RDFa.

Analogie générale

Je vois les TP comme une forêt : XML plante les arbres, XSD vérifie leur forme, XSLT les transforme en carte lisible, RDF relie les chemins entre eux, et RDFa ajoute des étiquettes invisibles pour les agents.

TP1 : XML, DTD, XPath, CSS et SVG

Le TP1 pose les bases : comprendre qu’un document XML est un arbre, corriger des fichiers mal formés, écrire une DTD, interroger un XML avec XPath et produire du SVG.

Ce que j’ai appris

  • Un XML bien formé possède une seule racine et des balises correctement imbriquées.
  • Un XML valide respecte une grammaire, ici une DTD.
  • XPath permet de poser des questions à l’arbre XML.
  • SVG montre qu’un langage XML peut aussi décrire une image.

Analogie : XML est comme un arbre : la racine tient tout, les branches organisent l’information, et les feuilles contiennent les données.

TP2 : XML Schema et XSLT

Ce TP approfondit XML : DTD et XSD servent à contrôler les données, puis XSLT transforme le document en HTML lisible.

Points importants

  • XSD permet de typer plus finement les données qu’une DTD : chaînes, identifiants, références, dates et structures complexes.
  • XSLT fonctionne avec des templates : quand un élément XML est reconnu, il est transformé en HTML.
  • Le CSS est appelé dans le HTML généré par XSLT, pas comme simple décoration dans le XML.

Analogie : XSD est le contrôleur qualité d’une bibliothèque, XSLT est le traducteur qui transforme les fiches internes en page lisible par le public.

Exemple de transformation

<xsl:template match="/">
  <html>
    <body>
      <xsl:apply-templates/>
    </body>
  </html>
</xsl:template>

TP3 : RDF, SPARQL et Corese

Dans ce TP, l’univers de La Compagnie des glaces est transformé en graphe RDF. On ne range plus seulement des données en arbre : on relie des personnages, des villes, des trains et des compagnies.

Modélisation RDF

Le modèle contient quatre classes principales : Personnage, Ville, Train et Compagnie. Les propriétés indiquent qui habite où, qui travaille pour qui, quel train relie quelles villes et à quelle compagnie il appartient.

:LucienBruk :habiteDans :NouvelleGeneve .
:LucienBruk :travaillePour :CompagnieDesGlaces .
:ExpressPolaire :relie :PortNord .

Analogie : RDF est comme un plan de métro : les stations sont les ressources, les lignes sont les propriétés, et SPARQL permet de demander quel trajet relie les éléments.

Requêtes SPARQL

  • Lister les personnages avec leur ville et leur compagnie.
  • Retrouver les trains par compagnie.
  • Classer les villes par température.
  • Identifier les conducteurs et les villes desservies.
SELECT ?nom ?ville ?compagnie
WHERE {
  ?p a glaces:Personnage ;
     glaces:nom ?nom ;
     glaces:habiteDans ?v ;
     glaces:travaillePour ?c .
}

TP4 : XML Schema, XSLT et Unicode

Ce TP mélange deux aspects : produire un XML fiable pour des débats du Parlement européen, puis comprendre comment les caractères Unicode sont encodés.

Partie XML Schema

Le schéma décrit des débats parlementaires avec des pays, mouvements politiques, députés, commissaires, séances et interventions. Les contraintes de clé assurent que les références pointent vers des éléments existants.

<xs:key name="cleDeputes">
  <xs:selector xpath=".//depute"/>
  <xs:field xpath="@id"/>
</xs:key>

Analogie : XML Schema joue le rôle du règlement d’entrée : chaque intervenant doit avoir une identité valide avant de prendre la parole dans le document.

Partie Unicode

L’application explique comment un caractère est représenté en UCS-4, UTF-8 et UTF-16 BE. Elle met en évidence les bits de structure et les bits de données.

  • UTF-8 est compact pour les caractères latins.
  • UTF-16 utilise des mots de 16 bits et parfois des paires de substitution.
  • L’encodage est essentiel pour un portfolio multilingue, notamment avec le japonais.

Analogie : Unicode est un passeport universel pour les caractères : chaque symbole reçoit une identité que les systèmes peuvent reconnaître.

TP5 : RDFa et Web sémantique

Le TP5 montre comment enrichir une page HTML avec des annotations RDFa pour qu’elle soit lisible par un humain et exploitable par un robot.

RDFa dans HTML

Les attributs about, typeof, property, rel et resource permettent de produire des triplets RDF à partir d’une page HTML.

<section about="#hasina" typeof="schema:Person foaf:Person">
  <h1 property="schema:name foaf:name">Hasina RAKOTOARISON</h1>
</section>

Analogie : RDFa est comme des post-it invisibles collés sur une page : l’humain voit le CV, mais l’agent sémantique lit les étiquettes.

LOD et extraction

Le TP utilise des vocabulaires comme FOAF, Dublin Core, Schema.org et DBpedia. Les liens owl:sameAs permettent d’aligner une ressource locale avec une ressource du Linked Open Data.

  • Extraction possible en Turtle.
  • Vérification avec un validateur RDFa ou OpenLink Structured Data Sniffer.
  • Requêtes SPARQL possibles sur DBpedia.
